프로젝트 사례

프로젝트 A – 서비스 대시보드

✔ 상태관리 구조 개선으로 중복 로직 제거 및 유지보수성 향상

역할: 프론트엔드 개발 (팀 5명)

문제

Redux 상태가 비대해지고, 비동기 로직과 UI 상태가 섞여 컴포넌트 수정 시 사이드 이펙트가 자주 발생.

해결

도메인 기준으로 상태를 분리하고, 전역 상태 / UI 상태 / 비즈니스 상태를 명확히 구분. 비동기 로직을 Service 레이어로 이동하여 책임을 분리.

결과

신규 기능 개발 속도 약 30% 개선, 상태 관련 버그 감소, 팀 내 공통 상태관리 패턴으로 정착.

프로젝트 B – 콘텐츠 관리 시스템

✔ 팀 내 상태 흐름 표준화 및 구조 문서화

역할: 프론트엔드 구조 설계 주도

문제

페이지별로 서로 다른 상태관리 방식 사용으로 팀원 간 코드 이해 비용 증가.

해결

공통 Store 구조와 네이밍 규칙 정의, 상태 흐름 다이어그램을 문서화하여 공유.

결과

신규 팀원의 온보딩 시간 단축, 코드 리뷰 효율 향상.

개발 철학

저는 단순히 기능을 구현하는 것보다, 왜 이 상태가 전역이어야 하는지, 어디까지가 컴포넌트의 책임인지를 고민합니다.

팀이 함께 이해하고 유지할 수 있는 구조를 만드는 것이 좋은 프론트엔드 개발자의 역할이라고 생각합니다.
